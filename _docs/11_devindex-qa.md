# 11. dev_index 통합 구축 가이드 & QA 체크리스트

**파일 경로:** `_docs/11_devindex-qa.md`
**체크리스트:** `deploy/QA_CHECKLIST.md`
**선행 문서:** 전체 산출물(00~10)

## 1. dev_index의 역할 재확인

dev_index는 개발·검수 기간에 개발자·기획자·고객 세 주체가 진척을 공유하는 단일 게이트다. 화면별 구현 상태를 매트릭스로 보여주고, 정적 미리보기·동작 테스트·수정 프롬프트 링크를 제공하며, 개발자 전용 도구(세션 주입·Mock 스위치)와 운영자 미리보기(가중치·비용)를 한 곳에 모은다.

## 2. 진척도 매트릭스 데이터 연동

매트릭스의 각 행은 `_docs/01_sitemap.md`와 `_docs/02_file-list.md`의 화면 ID ↔ `Screen` 키 ↔ React 컴포넌트 매핑을 기준으로 삼는다. 현재 `_publish`에는 `_MAPPING.md`와 화면별 HTML 파일이 없으므로, Codex는 이 매핑을 JSON(`_mock` 또는 `app/db`)으로 변환해 `GET /api/dev/screens` 형태로 제공한다. 각 행의 상태 배지는 ⚪대기 → 🟡코딩중 → 🔵로컬검증 → ●배포완료 4단계로, 상태가 바뀌면 색이 갱신된다.

## 3. 세 링크 연결 규칙

정적 미리보기 버튼(`preview-static`)은 `_publish/index.html`을 새 탭으로 연다(디자인 확인). 현재 화면별 HTML 경로가 없으므로 화면 구분은 `Screen` 키 또는 향후 추가할 쿼리/해시 라우팅 값으로 처리한다. 동작 테스트 버튼(`preview-live`)은 개발 서버의 실제 React 앱 진입점과 대상 `Screen` 키를 연다. 수정 프롬프트 버튼(`copy-prompt`)은 `PROMPT_MASTER.txt` 또는 `_publish/src/imports/`의 해당 화면 블록을 클립보드에 복사한다.

## 4. 개발자 도구 연동

가상 세션 주입(`inject-session`)은 `POST /api/dev/session-inject`를 호출해 일반/우수/게스트 토큰을 쿠키·로컬스토리지에 주입하고 사용자 화면으로 릴레이한다. Mock 스위치(`toggle-mock`)는 `PUT /api/dev/mock-switch`로 `USE_MOCK` 값을 토글한다. 이 두 도구는 `DEV_TOOLS_ENABLED=true`인 개발 환경에서만 노출되며, 운영 배포 시 자동 비활성화된다.

## 5. 인프라 상태 인디케이터 연동

상단 상태 바(로컬 DB·서버 DB·Nginx)는 `GET /api/dev/health`를 주기적으로 폴링해 청색(ONLINE)/적색(OFFLINE)을 표시한다. 적색일 때 'Show all databases' 가이드 툴팁을 노출한다. 운영자 미리보기의 가중치 슬라이더는 `policy_weights`와, 비용 표시는 `api_cost_logs`와 연동한다.

## 6. 구축 순서 요약

dev_index 구축은 다음 순서로 진행한다. 첫째, `_publish/src/app/App.tsx`의 `DevHub`와 `screenMap`을 기준으로 현재 화면 목록을 확정한다. 둘째, `app` 개발본으로 이관해 Codex가 매트릭스·세 링크·개발도구·상태바를 실제 API와 연결한다. 셋째, Mock 모드로 전 화면 전환과 도구가 동작하는지 검증한다. 넷째, 배포 후 동작 테스트 링크가 실제 서버의 React 앱 진입점과 대상 Screen 키를 가리키는지 확인한다.

# deploy/QA_CHECKLIST.md — 출시 전 검수 체크리스트

## A. 화면·디자인 (전 27화면 공통)
□ GNB/LNB·헤더·푸터가 모든 화면에서 통일되어 표시되는가
□ 타이포 위계(H1 28 / H2 22 / H3 18 / 본문 14 / 수치 24)가 일관된가
□ 컬러 토큰(Primary #0075d5, Error #D32F2F)만 사용됐는가
□ 모든 UI 텍스트가 한국어로 통일됐는가(약어만 영문)
□ 768px 이하에서 2분할→세로 스택, LNB→햄버거로 전환되는가
□ 랜딩 TTI 1.5초 이내, 이미지 WebP·레이지 로딩 적용됐는가

## B. 초급자 모드 흐름
□ 1~3단계 칩 선택 시 우측 프롬프트 대괄호가 실시간 치환되는가
□ 4단계 요약에서 textarea 직접 편집이 되고, 편집된 Raw Text가 전송되는가
□ '추천받기' 시 로딩 레이어 노출 후 result로 이동하는가
□ 가성비/추천/고성능 탭 전환 시 차트(fps-bar·price-pie)가 갱신되는가
□ 부품 변경 시 안전 대안 목록만 노출되는가
□ 호환성 위험 시 Error 표시 + 장바구니/결제 버튼이 disabled 되는가
□ 7초 내 일부 LLM 미응답 시 정상 모델만으로 결과가 나오는가(Fail-safe)

## C. 고급자 모드 흐름
□ 1~4단계 정밀 필터(제조사·세대·메모리·전력) 입력이 프롬프트에 반영되는가
□ 5단계 에디터에서 자유 편집 후 시스템 지시문이 자동 결합 전송되는가
□ 하드 제약(브랜드) 위반 응답이 Drop되는가
□ 전력 마진 부족·물리 치수 초과 견적이 제외되는가
□ result-expert에 1% Low FPS·5대 호환성 매트릭스가 표시되는가
□ 고사양 GPU 스왑 시 연쇄 변경 안내가 뜨고, 승인 시 Error가 자동 해제되는가

## D. 관리자·분석
□ 상품 검색이 카테고리+상태+키워드로 동작하고 50건 페이징되는가
□ 상품 수정 시 AI 연산 필드(소켓·전력·치수) 누락이 저장 차단되는가
□ CSV 업로드가 신규 Insert/기존 Update로 처리되고 오류 행이 분리되는가
□ 품절/단종 전환 시 1초 이내 추천 풀·스왑 대안에서 제외되는가
□ 대시보드 3사 비용·추이 차트가 표시되는가
□ Rate Limit 설정 후 한도 초과 시 429가 반환되는가
□ 인기 키워드·탈락 부품 TOP10·전환 퍼널 차트가 렌더링되는가

## E. 비용·보안 (최우선)
□ Mock 모드(USE_MOCK=true)에서 외부 LLM이 호출되지 않는가(비용 0)
□ 일일 비용 임계치 도달 시 Circuit Breaker가 외부 호출을 차단하는가
□ Rate Limit·Circuit Breaker가 외부 호출 '이전'에 작동하는가
□ API Key가 소스·Git·프론트·관리자 화면 어디에도 평문 노출되지 않는가
□ .env가 Git에 커밋되지 않았는가
□ 프롬프트 전송 전 개인정보(PII)가 마스킹되는가
□ DEV_TOOLS_ENABLED=false로 개발 전용 API가 운영에서 막혔는가
□ dev_index 게이트가 외부에 무방비 노출되지 않았는가

## F. 인프라·배포
□ Nginx가 정적은 직접 서빙, /api만 프록시하는가
□ PM2로 백엔드가 상시 구동되고 재부팅 시 자동 시작되는가
□ DB 복합 인덱스로 logs 10만건 0.5초·상품 검색 0.3초를 만족하는가
□ deploy.sh 한 줄로 배포가 완료되는가
□ dev_index 인프라 상태 바(로컬DB·서버DB·Nginx)가 정상 표시되는가
□ 단방향 이관(_publish React 앱 → app 개발본)이 완료되고 화면 ID/Screen 키 매핑에 기록됐는가

## G. 단방향 이관 무결성
□ _publish/ 원본이 개발 중 수정되지 않았는가
□ 모든 화면의 Screen 키·컴포넌트·이벤트 핸들러·차트가 연결됐는가
□ 디자인 변경이 _publish 재생성→개발본 재이관 절차로만 이뤄졌는가

## 전체 산출물 최종 정리

지금까지 만든 산출물을 한눈에 정리하면 다음과 같습니다.

기획·정의로 `00_glossary`(헌장·용어), `01_sitemap`(메뉴 구성도), `02_file-list`(파일 구조서), `03_userflow`(플로우·예외)를 만들었고, 설계로 `04_wireframe/`(와이어 3종), `05_design-guide`(디자인 시스템), `06_db-erd`(DB 설계), `07_api-spec`(API 명세), `08_architecture`(아키텍처)를 만들었습니다. 협업 규격으로 `STITCH_RULES`(Stitch 규격), `CODEX_HANDOFF`(Codex 인계), `stitch-prompts/batch1~5`(27화면 생성 프롬프트), `PROMPT_MASTER.txt`(프롬프트 마스터)를 만들었고, 데이터·배포로 `09_mock-data`(테스트·목 데이터), `10_infra-deploy`(인프라·배포), `11_devindex-qa` + `QA_CHECKLIST`(게이트·검수)를 완성했습니다.

이제 실제 작업 순서는 이렇습니다. 먼저 현재 `_publish/`의 Vite/React SPA 구조와 `App.tsx`의 Screen 매핑을 기준으로 개발본 편입 방식을 확정합니다. 그다음 Codex로 DB 마이그레이션과 백엔드(오케스트레이터·검증·비용통제)를 구축하고, `_publish` React 화면을 `app`으로 이관하며 로직을 연결합니다. 마지막으로 Mock 모드로 QA 체크리스트를 통과시킨 뒤 개발 서버에 배포합니다.
