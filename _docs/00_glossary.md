# 00. 프로젝트 헌장 & 용어 사전

**파일 경로:** `_docs/00_glossary.md`
**문서 버전:** Ver 1.0
**적용 범위:** 팝콘PC AI 조립PC 추천 및 구매 시스템 전체

## 1. 프로젝트 헌장 (Project Charter)

### 1.1 프로젝트 개요
팝콘PC의 기존 쇼핑몰과 연동되는 **AI 기반 조립PC 추천·구매 시스템**을 구축한다. 사용자는 하드웨어 지식 수준에 따라 초급자 모드 또는 고급자 모드로 진입하며, 3대 대형 LLM(Google Gemini, OpenAI ChatGPT, Anthropic Claude)이 병렬로 견적을 생성하고, 내부 검증 엔진이 팝콘PC 실재고·호환성·마진을 대조해 최적안을 확정한다. 관리자는 백오피스에서 비용·정책·상품을 통제한다.

### 1.2 비즈니스 목표
이 시스템은 "AI 영업사원"이라는 콘셉트로, 하드웨어를 모르는 고객도 1분 내 맞춤 견적을 받게 하여 구매 전환율을 높이고, 동시에 외부 LLM API 비용을 통제하며 마진을 최적화하는 것을 목표로 한다.

### 1.3 프로젝트 기간 및 인프라
프로젝트 기간은 2026-04-01부터 2026-06-30까지이며, 개발 서버 인프라는 Ubuntu 24.04 LTS, Nginx, PostgreSQL, Node.js로 구성된다.

### 1.4 관여 주체 및 역할 (RACI 요약)

| 주체 | 주요 역할 | 사용 도구 | 작업 위치 |
|------|----------|----------|----------|
| 기획자 | 요구사항 확정, 화면 디자인 지시, 진척 관리 | Google Stitch, VS Code, GitHub | 로컬 PC |
| 개발자 | 프로그램 소스 개발, DB·API·배포 | Codex AI, VS Code | 로컬 PC → 개발 서버 |
| 고객 | 진척 확인, 화면 검수, 피드백 | 웹 브라우저 | React SPA의 `dev-hub` 게이트 |
| 운영자(관리자) | 비용·정책·상품 통제 | 백오피스 관리자 화면 | 배포된 운영 화면 |

### 1.5 핵심 작업 원칙 (필독)
모든 주체는 다음 5개 원칙을 준수한다.

첫째, **단방향 이관 원칙**이다. `_publish/`에 생성된 와이어프레임 기반 정적 React 소스는 원본으로 보존하며 절대 직접 수정하지 않는다. 개발은 `_publish/` 구조를 기준으로 `app/`에 편입한 사본 또는 분리된 개발본에서만 진행한다.

둘째, **도구 역할 경계 원칙**이다. Stitch/Figma Make 계열 산출물은 화면 React 소스·CSS·기본 인터랙션까지만 담당하고, Codex는 백엔드·DB·API·실데이터 연동·운영 상태관리를 담당한다.

셋째, **화면 ID 단일 식별 원칙**이다. 현재 `_publish` 기준 모든 화면은 요구사항 ID(예: FR-LND-010), `Screen` 키(예: `landing`), React 컴포넌트명(예: `Landing`)으로 식별한다. 실제 매핑은 `_docs/01_sitemap.md`와 `_docs/02_file-list.md`의 표를 기준으로 관리한다.

넷째, **비용 안전 원칙**이다. 개발·테스트 중에는 Mock API 스위치를 기본 활성화하여 외부 LLM 호출 비용을 0으로 유지하고, 실제 호출은 명시적으로 전환했을 때만 발생한다.

다섯째, **보안 격리 원칙**이다. 유료 API Key는 어떤 경우에도 소스나 Git에 평문으로 두지 않으며 서버의 `.env`에만 저장한다.

## 2. 용어 사전 (Glossary)

### 2.1 도메인·모드 용어

| 용어 | 정의 |
|------|------|
| 초급자 모드 (Easy Mode / Beginner) | 하드웨어 지식이 부족한 일반 소비자용. 용도·예산·감성 옵션만으로 견적. 4단계(용도→예산→옵션→요약). 내부 코드값 `beginner` |
| 고급자 모드 (Expert Mode) | 하드웨어 전문 지식 보유자용. 제조사 고정·세대·전력 마진 등 5단계 정밀 필터. 내부 코드값 `expert` |
| 관리자 허브 (Admin Hub) | 운영자가 비즈니스 지표, 상품 마스터, 재고 상태, 가격 정책, 추천 가중치, 마케팅 분석, AI 비용·Rate Limit을 통합 제어하는 백오피스. V3.0 기준 우선순위는 비즈니스 대시보드 → 상품 마스터 및 재고 제어 → 가격 정책 및 가중치 → 마케팅 분석 → 시스템 인프라 순서다. |
| 상품 마스터 및 재고 제어 | 관리자 사이트맵 V3.0에서 AI 비용 관리보다 상위로 격상된 핵심 운영 레이어. `products` 및 `product_specs`를 기반으로 상품 검색, AI 호환성 필드 수정, 품절/단종 토글, CSV 업서트를 처리한다. |
| 운영자 메인 컨트롤 타워 | `adm-dashboard` 화면. 실시간 AI 최종 견적 피드, 특가 CTR, 유저 키워드 트렌드, 가격동향, AI 비용 미니 위젯을 함께 보여주는 관리자 첫 화면이다. |
| 게스트/비회원 | 로그인 없이 유입된 사용자. 임시 UID 발급 대상 |

### 2.2 기술·아키텍처 용어

| 용어 | 정의 |
|------|------|
| 오케스트레이터 (Orchestrator) | 3사 LLM API를 병렬 비동기로 호출·취합하는 백엔드 모듈. `Promise.allSettled` 기반 |
| 프롬프트 빌더 (Prompt Builder) | 사용자가 조건을 선택할 때마다 우측 패널에서 대괄호 `[]` 키워드를 실시간 치환해 자연어 주문서를 조립하는 프론트 컴포넌트 |
| Raw Text | 사용자가 최종 요약 단계에서 직접 편집한 원시 프롬프트 문자열. 추천 요청의 최종 페이로드 |
| Fail-safe | 7초 타임아웃 내 미응답·JSON 깨짐 모델을 탈락시키고 정상 모델만으로 연산을 완결하는 구조 |
| Mock API Switch | 외부 LLM 호출을 차단하고 사전 저장 JSON을 반환하는 비용 통제 스위치 |
| Circuit Breaker | 일일 비용 임계치(예: $50) 도달 시 외부 통신을 강제 차단하는 셧다운 트리거 |
| Rate Limit | 회원 등급·IP별 일일 AI 호출 횟수 제한. 초과 시 429 반환. Redis 카운터 사용 |
| Upsert | CSV 일괄 반영 시 신규는 Insert, 기존은 Update하는 처리. PK는 `자체상품코드` |
| 하드 제약 조건 (Hard Constraints) | 고급자가 지정한 필수 브랜드·규격. 위반 모델은 점수 계산 없이 즉시 Drop |
| 연쇄 스왑 (Chain Swap) | 부품 변경으로 호환성 깨질 때 관련 부품(예: 파워)을 함께 교체 제안하는 흐름 |
| 안전 대안 목록 | 현재 구성과 호환성 100% 검증 통과한 교체 후보만 노출하는 목록 |
| 실시간 품절 차단 토글 | 관리자가 특정 상품 상태를 `품절` 또는 `단종`으로 변경하는 즉시, 1초 이내에 AI 추천 후보군과 부품 스왑 안전 대안 목록에서 해당 상품을 전역 제외하는 관리자 기능. |
| AI 호환성 정형 필드 | 추천·검증 엔진이 참조하는 `product_specs`의 소켓, 칩셋, 메모리 규격, TDP, 정격 W, GPU 길이 mm, 케이스 장착 한계 mm, 쿨러 해소 TDP 등의 필드. |
| 가격 정책 | CPU, GPU, 메모리, SSD, 파워, 케이스 등 카테고리별 기본 마진율과 가격 가산 정책을 관리자가 조정하는 백오피스 정책 레이어. |
| 추천 가중치 밸브 | 재고소진 비중, 마진극대 비중, 가성비 비중의 합이 100%가 되도록 조정하는 관리자용 추천 엔진 정책 슬라이더. |
| CTR 리포트 | 특가상품, 추천 부품, 배너, 상세 조회 등 노출 대비 클릭률을 분석하는 마케팅 지표 리포트. |
| AI 비용 미니 위젯 | 메인 관리자 대시보드 우측 하단에 축소 배치되는 Gemini, ChatGPT, Claude 누적 비용 및 Circuit Breaker 상태 표시 UI. |

### 2.4 화면 ID 명명 규칙

사용자 화면은 기존과 동일하게 `FR-[도메인]-[일련번호]` 형식을 따른다.  
도메인 코드는 LND(랜딩), BEG(초급자), EXP(고급자), RES(결과 공통), DEV(개발 허브)를 사용한다.

관리자 화면은 V3.0 사이트맵부터 실제 운영 우선순위를 더 명확히 표현하기 위해 별도 관리자 도메인 ID를 사용한다.

| 관리자 도메인 | 의미 | 예시 |
|---------------|------|------|
| ADM-DSH | 관리자 메인 대시보드 | `ADM-DSH-010` |
| ADM-PRD | 상품 마스터 및 재고 제어 | `ADM-PRD-010` |
| ADM-CSV | CSV 업서트 | `ADM-CSV-010` |
| ADM-POL | 가격 정책 및 추천 가중치 | `ADM-POL-010`, `ADM-POL-020` |
| ADM-ANA | 마케팅 및 사용자 행동 분석 | `ADM-ANA-010` |
| ADM-SYS | 시스템 인프라 및 비용 제어 | `ADM-SYS-010` |
| ADM-OPS | 운영자 및 권한 관리 | `ADM-OPS-010` |
| DEV-HUB | 개발·검수 허브 | `DEV-HUB-010` |


### 2.5 상태 플래그 표준값

화면 구현 상태는 4단계로 관리한다: `대기` → `바이브 코딩 중(로컬)` → `로컬 검증 완료` → `개발 서버 배포 완료(Nginx)`. 상품 상태값은 CSV 기준으로 `판매중`, `품절`, `단종`, `삭제대기`를 사용하며, 추천 엔진은 `판매중` 상품만 후보로 삼는다.
