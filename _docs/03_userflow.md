이 문서는 사용자가 화면을 어떻게 이동하는지(정상 흐름)와, 무엇이 잘못될 수 있는지(예외 흐름)를 모두 정리한 것입니다. 특히 7초 타임아웃, 호환성 Error, 연쇄 스왑, Rate Limit, 비용 차단 같은 예외 처리가 Codex 개발의 핵심이므로 빠짐없이 명세했습니다.

# 03. 사용자 플로우 & 예외 시나리오

**파일 경로:** `_docs/03_userflow.md`
**문서 버전:** Ver 1.0
**선행 문서:** `00_glossary.md`, `01_sitemap.md`, `02_file-list.md`

## 1. 공통 진입·인증 플로우

### 1.1 정상 흐름
사용자가 `_publish/index.html` 또는 개발본 앱 진입점에 들어오면 React SPA의 `landing` 화면이 렌더링되고 GNB와 히어로 카피, 초급자·고급자 분기 카드가 노출된다. 로그인이 필요한 시점(견적 보관, 장바구니)에 [로그인/회원가입]을 누르면 `auth-modal` 화면이 호출되고, 인증 성공 시 JWT 토큰 또는 암호화 세션 키가 로컬 스토리지·쿠키에 바인딩된다.

### 1.2 예외 흐름
| 예외 | 발생 조건 | 처리 |
|------|----------|------|
| 비로그인 견적 진행 | 게스트가 추천까지 진행 | 임시 UID 발급, 결과·상세까지 허용. 장바구니/결제 단계에서만 로그인 유도 모달 |
| SSO 인증 실패 | 외부 회원 DB 응답 오류 | 모달 내 오류 메시지, 게스트 모드로 계속 진행 옵션 제공 |
| 세션 만료 | JWT 유효시간 초과 | 다음 보호 액션 시 재인증 모달 재호출, 입력값은 세션 스토리지에 보존 |

## 2. 초급자 모드 플로우

### 2.1 정상 흐름
```
landing [초급자 카드]
  → beg-step1      (용도 칩 선택 + 게임 시 하위 소프트웨어 태그)
  → beg-step2      (양방향 슬라이더 또는 간편 예산 칩)
  → beg-step3      (#화이트감성 등 감성 옵션 칩)
  → beg-step4      (자동 조립된 프롬프트 + 자유 편집 허용)
       → ['추천받기'] → 로딩 레이어("3대 AI가 조율 중...")
            → 백엔드 오케스트레이터 (3사 병렬 호출 + 4단계 검증)
                 → beg-result (가성비/추천/고성능 3종 탭, 차트)
                      → [부품 변경] beg-detail (안전 대안 목록)
                           → [장바구니 담기] → 외부 쇼핑몰 결제 레일
```

각 단계에서 좌측 입력(60%)을 변경할 때마다 우측 프롬프트 패널(40%)의 대괄호 키워드가 실시간 치환된다. 4단계 요약에서는 Readonly가 해제되어 사용자가 직접 편집할 수 있고, '추천받기' 시 UI 칩 값이 아니라 **편집된 Raw Text 전체**가 페이로드로 전송된다.

### 2.2 예외 흐름
| 예외 ID | 상황 | 처리 방식 |
|---------|------|----------|
| EX-BEG-01 | 7초 내 일부 LLM 미응답/JSON 깨짐 | 해당 모델만 Drop, 정상 모델 결과로 연산 계속(Fail-safe) |
| EX-BEG-02 | 3사 모두 타임아웃/실패 | "잠시 후 다시 시도" 안내 레이어, 입력값 보존, 재시도 버튼 |
| EX-BEG-03 | 추천 부품이 모두 품절/단종 | 검증 단계에서 후보 제외, 차순위안 노출. 후보 없으면 예산 상향 안내 |
| EX-BEG-04 | 부품 스왑으로 물리 크래시 우려 | 호환성 상태 `Error(#D32F2F)` 전환, 장바구니/결제 버튼 `disabled` 락 |
| EX-BEG-05 | 예산 초과 견적만 생성됨 | 업셀링/다운셀링 가이드 모달로 대안 제시 |
| EX-BEG-06 | 일일 호출 제한 초과(회원) | 429 반환, "오늘 이용 한도 초과" 안내, AI 미호출 |

## 3. 고급자 모드 플로우

### 3.1 정상 흐름
```
landing [고급자 카드]
  → exp-step1      (밸런스 가중치 + CPU/GPU 제조사 고정)
  → exp-step2      (세대 선택 + 오버클럭 K/X 라인업)
  → exp-step3      (DDR4/5, 용량, NVMe PCIe 4.0/5.0)
  → exp-step4      (전력 마진율 슬라이더 + 쿨링 형태)
  → exp-step5      (전문가 프롬프트 전체 편집 개방)
       → ['추천받기'] → 백엔드 (병렬 호출 + 하드 제약 필터 + 물리 시뮬레이션)
            → exp-result (1% Low FPS, 5대 호환성 매트릭스)
                 → [부품 변경] exp-detail
                      → [연쇄 스왑 승인] → [장바구니] → 외부 결제 레일
```

### 3.2 예외 흐름
| 예외 ID | 상황 | 처리 방식 |
|---------|------|----------|
| EX-EXP-01 | LLM이 하드 제약(브랜드/규격) 위반 응답 | 점수 계산 없이 즉시 Drop |
| EX-EXP-02 | 제안 파워 정격 < (합산 TDP×(1+마진율)) | 해당 견적 강제 제외 |
| EX-EXP-03 | 쿨러 해소 TDP < CPU 최대 발열 TDP | 쿨링 마진 부족, 탈락 |
| EX-EXP-04 | GPU 길이 > 케이스 장착 한계(1mm라도 초과) | 물리 충돌, 원천 제거 |
| EX-EXP-05 | 고사양 GPU 스왑으로 파워 부족 | `Error` 점등 + [연쇄 변경 해결책] 데이터 반환 |
| EX-EXP-06 | 연쇄 스왑 승인 | 파워 자동 동시 교체, `Error` 상태 자동 Clear |
| EX-EXP-07 | 모든 제약 만족 견적 없음 | 가장 근접한 차선안 + 어떤 제약이 충돌했는지 사유 표기 |

## 4. 관리자 플로우

### 4.1 정상 흐름
관리자는 로그인 후 `adm-dashboard`에서 3사 API 실시간 토큰·달러 비용을 확인한다. 거기서 `adm-monitoring`(트래픽), `adm-rate-limit`(이용 제한)로 이동해 비용을 통제하고, `adm-master`에서 상품을 검색·수정하며, `adm-csv`로 대량 업서트를 수행하고, `adm-keywords`·`adm-swap-report`·`adm-funnel`에서 분석 화면을 확인한다.

### 4.2 예외·트리거 흐름
| 예외 ID | 상황 | 처리 방식 |
|---------|------|----------|
| EX-ADM-01 | 일일 비용 임계치($50 등) 도달 | Circuit Breaker 작동, 외부 통신 차단, 사용자에 점검 레이어 |
| EX-ADM-02 | 회원이 일일 호출 한도 초과 | Redis(또는 인메모리) 카운터 대조 → 429 반환, AI 미호출 |
| EX-ADM-03 | 상품 상태를 품절/단종으로 변경 | 1초 이내 추천 풀·스왑 대안 목록에서 완전 제외 |
| EX-ADM-04 | CSV 업로드 중 형식 오류 행 | 정상 행은 처리, 오류 행은 리포트로 분리 반환 |
| EX-ADM-05 | 필수 연산 필드(소켓/전력/치수) 누락 | 저장 전 Validation 차단, 입력 요구 |

## 5. 개발 게이트(dev_index) 플로우

개발·검수 기간에 세 주체는 React SPA 내부의 `dev-hub` 화면으로 진입한다. 기획자·고객은 `_publish/index.html` 기반 정적 미리보기로 디자인을 확인하고, 개발자는 개발 서버에서 동작 테스트를 확인한다. 현재 `_publish`에는 화면별 HTML 파일이 없으므로 화면 추적은 `Screen` 키와 React 컴포넌트명을 기준으로 한다. 개발 전용 보조 기능으로 가상 세션 주입기(일반회원/우수회원/게스트)와 Mock API 스위치가 있어, 외부 LLM 비용 없이 화면·차트 렌더링을 검증한다.

## 6. 전역 상태 전이 다이어그램 (추천 요청)

```
[입력 완료] 
   │ '추천받기'
   ▼
[로딩] ──(7초 타임아웃 감시)──┐
   │ 정상 응답 도착            │ 전부 실패
   ▼                          ▼
[검증 엔진]                 [재시도 안내]
   │ 후보 있음   │ 후보 없음
   ▼            ▼
[결과 렌더링]  [대안/예산조정 안내]
   │ 부품 스왑
   ▼
[호환성 검증] ──위반──→ [Error 락(버튼 disabled)] ──(연쇄스왑 승인)──→ [자동 해제]
   │ 정상
   ▼
[장바구니 → 외부 결제 레일]
```

## 7. 예외 처리 우선순위 원칙

여러 예외가 동시에 발생할 경우 처리 우선순위는 다음과 같다. 첫째, 비용·보안 관련(Circuit Breaker, Rate Limit)이 최우선으로, 외부 호출 자체를 막는다. 둘째, 물리·전기 호환성 위반은 결제를 막는다(오발주 방지). 셋째, 재고·예산 관련은 대안 제시로 처리한다. 넷째, 네트워크 지연은 Fail-safe로 흡수한다. 이 순서는 "돈이 새거나 조립 불가능한 제품이 팔리는 것"을 가장 먼저 차단한다는 비즈니스 원칙을 반영한다.
